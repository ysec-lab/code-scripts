## General Meeting & Collaboration Terms

| Term | Definition | Example |
|---|---|---|
| <a id="offline"></a>"Let's take this offline" | Let's discuss this outside of the current meeting, as it's not relevant to everyone. | "That's a good point, but let's take this offline so we can finish the stand-up." |
| <a id="parking-lot"></a>"Parking Lot" | A place to "park" topics that are important but not relevant to the current discussion. | "This is a bit off-topic. Let's put it in the parking lot and address it at the end of the meeting." |
| <a id="sync-up"></a>"Sync up" | To get on the same page and share information. | "I need to sync up with the design team about the new mockups." |
| <a id="deep-dive"></a>"Deep Dive" | To investigate a topic in great detail. | "This afternoon, we're having a deep dive session on the new architecture proposal." |
| <a id="circle-back"></a>"Circle back" | To return to a topic at a later time. | "I don't have the answer right now, but I will circle back with you by the end of the day." |
| <a id="low-hanging"></a>"Low-hanging fruit" | The easiest tasks to complete that still provide value. | "Let's tackle the low-hanging fruit first to show some quick progress." |
| <a id="boil-ocean"></a>"Boil the ocean" | To try to do too much at once; to make a project unnecessarily difficult. | "A full redesign is not feasible right now; we can't boil the ocean." |
| <a id="shave-yak"></a>"Shave the yak" | To get sidetracked by a series of small, seemingly necessary tasks. | "I just wanted to change a button color, but I've been shaving the yak for hours setting up a new dev environment." |
| <a id="happy-path"></a>"Happy Path" | The default scenario with no errors or exceptions. | "The happy path works, but what happens when a user enters invalid data?" |
| <a id="edge-case"></a>"Edge Case" | A problem that occurs only at an extreme operating parameter. | "We need to test for edge cases, like a user uploading a 10 GB file." |
| <a id="same-page"></a>"On the same page" | Having a shared understanding. | "Before we proceed, let's make sure everyone is on the same page." |
| <a id="hard-stop"></a>"Hard stop" | A definitive end time for a meeting. | "I have another meeting at 11, so I have a hard stop." |
| <a id="action-item"></a>"Action Item (AI)" | A specific task assigned to a person to be completed by a certain date. | "The action item from this meeting is for John to investigate the performance issue." |
| <a id="bandwidth-personal"></a>"Bandwidth" (personal) | A person's capacity to take on more work. | "I'd like to help, but I don't have the bandwidth right now." |
| <a id="out-of-the-box"></a>"Out of the box" | Thinking creatively and unconventionally. | "We need some out of the box ideas to solve this problem." |
| <a id="drill-down"></a>"Drill down" | To look at something in more detail. | "Can we drill down into the Q3 sales numbers?" |
| <a id="piggyback"></a>"Piggyback" | To add a related thought to what someone else has just said. | "If I can piggyback on what Sarah said, I also think the documentation is lacking." |
| <a id="double-click"></a>"Double-click on that" | To explore a topic in more detail. | "That's an interesting point. Let's double-click on that for a moment." |
| <a id="ramp-up"></a>"Ramp up" | The process of getting a new team member up to speed. | "It will take a few weeks for the new hire to ramp up on our codebase." |
| <a id="poc"></a>"POC (Proof of Concept)" | A small project to test the feasibility of an idea. | "We're building a quick POC to see if this new library will meet our needs." |
| <a id="sanity-check"></a>"Sanity Check" | A quick review to see if a decision or approach makes sense. | "Before I commit this, can someone give my code a quick sanity check?" |
| <a id="post-mortem"></a>"Post-mortem" | A meeting after a project or incident to discuss what happened. | "We're holding a post-mortem tomorrow to discuss the server outage." |
| <a id="blocking-tackling"></a>"Blocking and tackling" | Doing the fundamental, essential work. | "Forget the fancy features for a moment; we need to focus on the basic blocking and tackling of bug fixes." |
| <a id="high-level"></a>"At a high level" | A general overview without going into detail. | "At a high level, the project is on track, but there are some risks we need to manage." |
| <a id="in-the-weeds"></a>"In the weeds" | To be too focused on the minor details. | "We're getting too in the weeds with font sizes. Let's focus on the overall user workflow first." |

## Technical & IT Terms

| Term | Definition | Example |
|---|---|---|
| <a id="api"></a>API (Application Programming Interface) | A set of rules that allows different software applications to communicate. | "The mobile app is failing because the backend API is not responding." |
| <a id="cicd"></a>CI/CD (Continuous Integration/Continuous Deployment) | The practice of automating the build, test, and deployment of code. | "The build failed in our CI/CD pipeline; someone needs to investigate." |
| <a id="build"></a>Build | The process of compiling source code into a runnable application. | "The latest build is ready for deployment to the QA environment." |
| <a id="deployment"></a>Deployment | The process of releasing new code to an environment. | "The deployment to production is scheduled for 10 PM tonight." |
| <a id="environment"></a>Environment (Env) | A server or set of servers used for a specific purpose (e.g., Dev, QA, Staging, Prod). | "The bug is only reproducible in the QA environment." |
| <a id="git"></a>Git | A version control system for tracking changes in code. | "Make sure you commit your latest changes to Git." |
| <a id="branch"></a>Branch | A parallel version of the codebase, allowing for independent development. | "Create a new branch for the feature you're working on." |
| <a id="merge"></a>Merge | The process of combining changes from different branches. | "I need to merge the latest changes from the main branch into my feature branch." |
| <a id="pr"></a>Pull Request (PR) / Merge Request (MR) | A request to merge one branch into another, often triggering a code review. | "I've submitted the Pull Request for my changes; can someone please review it?" |
| <a id="code-review"></a>Code Review | The process of having other developers examine code for quality and correctness. | "Please don't merge your code until it has passed a Code Review." |
| <a id="master"></a>Master / Main | The primary branch in a Git repository. | "We should never commit directly to the main branch." |
| <a id="hotfix"></a>Hotfix | A critical bug fix that needs to be deployed to production immediately. | "We need to deploy a hotfix to resolve the login issue on the production server." |
| <a id="feature-flag"></a>Feature Flag / Toggle | A technique to turn features on or off without deploying new code. | "Let's release the new feature under a feature flag so we can enable it for testers first." |
| <a id="unit-test"></a>Unit Test | A test for a small, isolated piece of code. | "My code is done, I just need to write the unit tests for it." |
| <a id="integration-test"></a>Integration Test | A test to verify that different parts of the application work together. | "The integration test failed because the API and the database are out of sync." |
| <a id="e2e"></a>End-to-End (E2E) Test | A test that simulates a real user workflow from start to finish. | "The E2E test for the checkout process is failing at the payment step." |
| <a id="regression"></a>Regression | A bug where a feature that used to work stops working. | "The latest deployment caused a regression in the search functionality." |
| <a id="smoke-test"></a>Smoke Test | A quick set of tests to ensure the most critical functions of an application are working. | "After every deployment, we run a quick smoke test to ensure the site is up." |
| <a id="qa"></a>QA (Quality Assurance) | The process of ensuring the software meets quality standards. | "I'm moving this ticket to the QA column for testing." |
| <a id="uat"></a>UAT (User Acceptance Testing) | Testing by end-users to verify the software meets their needs. | "We need to schedule a UAT session with our business stakeholders next week." |
| <a id="devops"></a>DevOps | A culture and set of practices that combines software development and IT operations. | "Our DevOps team is working on automating the server provisioning process." |
| <a id="frontend"></a>Frontend | The part of the application that the user interacts with (the client-side). | "The bug is in the frontend; the button isn't firing the correct event." |
| <a id="backend"></a>Backend | The server-side of the application that handles logic and data. | "I need to add a new endpoint to the backend to support this feature." |
| <a id="db"></a>Database (DB) | Where the application's data is stored. | "We need to add a new column to the users table in the database." |
| <a id="latency"></a>Latency | The delay before a transfer of data begins following an instruction. | "Users in Australia are experiencing high latency when accessing our servers." |
| <a id="bandwidth-tech"></a>Bandwidth (Tech) | The maximum rate of data transfer across a given path. | "The image uploads are failing because we're exceeding our server's bandwidth." |
| <a id="scalability"></a>Scalability | The ability of a system to handle a growing amount of work. | "We need to architect this service for scalability to handle holiday traffic." |
| <a id="microservices"></a>Microservices | An architectural style that structures an application as a collection of small, independent services. | "The payment processing is handled by our dedicated 'Payments' microservice." |
| <a id="monolith"></a>Monolith | A single, unified application, as opposed to microservices. | "We're slowly breaking down our old monolith into microservices." |
| <a id="containerization"></a>Containerization (e.g., Docker) | Packaging an application and its dependencies into a standardized unit. | "Containerization makes it easy to run the application consistently across different environments." |
| <a id="orchestration"></a>Orchestration (e.g., Kubernetes) | Automating the management and deployment of containerized applications. | "We use Kubernetes for orchestration to manage our fleet of microservices." |
| <a id="legacy"></a>Legacy Code | Old, often poorly written code that is difficult to work with. | "I'm hesitant to make changes because this part of the system is full of legacy code." |
| <a id="scope-creep"></a>Scope Creep | When new requirements are added after the initial scope has been defined. | "This is classic scope creep; the client is asking for three new features that weren't in the original story." |
| <a id="gold-plating"></a>Gold Plating | Adding extra features that were not requested by the client. | "Let's avoid gold plating this feature; just build what the acceptance criteria asks for." |
| <a id="mvp"></a>MVP (Minimum Viable Product) | A version of a product with just enough features to be usable by early customers. | "For the first release, let's just focus on the MVP and get feedback." |
| <a id="yagni"></a>YAGNI (You Ain't Gonna Need It) | The principle of not adding functionality until it is necessary. | "Should we add caching here? Let's apply YAGNI for now and add it later if performance is an issue." |
| <a id="dry"></a>DRY (Don't Repeat Yourself) | The principle of reducing repetition in code. | "You've written the same function in three different places. Let's make it DRY by creating a shared utility." |
| <a id="kiss"></a>KISS (Keep It Simple, Stupid) | The principle of designing systems to be as simple as possible. | "The proposed solution is too complex. Let's remember the KISS principle." |
| <a id="band-aid"></a>Band-Aid Fix | A quick, temporary solution that doesn't address the root cause of a problem. | "This is just a band-aid fix to get the system running; we need a proper solution next sprint." |
| <a id="rca"></a>Root Cause Analysis (RCA) | A method of problem-solving used to identify the fundamental cause of an issue. | "We need to conduct a Root Cause Analysis to understand why the server crashed." |

## Core Scrum Roles & Events

| Term | Definition | Example |
|---|---|---|
| <a id="scrum-master"></a>Scrum Master | The facilitator for the team, a servant-leader who removes impediments. | "I'm blocked by a license issue. Can the Scrum Master help escalate this?" |
| <a id="po"></a>Product Owner (PO) | The voice of the customer; responsible for the product backlog and prioritizing work. | "We need clarification on the requirements. Let's ask the Product Owner to join the call." |
| <a id="dev-team"></a>Development Team | The cross-functional group of people who do the actual work of building the increment. | "The Development Team has estimated that this feature will take the entire sprint." |
| <a id="stakeholder"></a>Stakeholder | Anyone with an interest in the product (e.g., users, management, clients). | "We need to invite the key stakeholders to the Sprint Review to get their feedback." |
| <a id="sprint"></a>Sprint | The fixed time-box (usually 1-4 weeks) where the team works to complete a set amount of work. | "We have three more days left in the current sprint." |
| <a id="sprint-planning"></a>Sprint Planning | The meeting at the start of a sprint to decide what will be built. | "Let's finalize the scope for the next two weeks in our Sprint Planning session tomorrow." |
| <a id="daily-standup"></a>Daily Stand-up / Daily Scrum | The 15-minute daily meeting to sync up on progress, plans, and impediments. | "I'll provide a full update during tomorrow's Daily Stand-up." |
| <a id="sprint-review"></a>Sprint Review | The meeting at the end of a sprint to demo the work that was completed. | "Make sure the demo environment is ready for the Sprint Review on Friday." |
| <a id="sprint-retro"></a>Sprint Retrospective (Retro) | The meeting at the end of a sprint to discuss what went well, what didn't, and how to improve. | "Let's bring up the communication issues in our next Sprint Retrospective." |
| <a id="refinement"></a>Backlog Refinement / Grooming | The ongoing process of reviewing and estimating items in the product backlog. | "We have Backlog Refinement scheduled for Wednesday to estimate the new user stories." |

## Scrum Artifacts & Concepts

| Term | Definition | Example |
|---|---|---|
| <a id="product-backlog"></a>Product Backlog | The master list of all desired features, enhancements, and fixes for the product. | "The PO has just added three new features to the Product Backlog." |
| <a id="sprint-backlog"></a>Sprint Backlog | The subset of product backlog items selected for a specific sprint. | "Once we commit to the work, we can't add anything else to the Sprint Backlog." |
| <a id="increment"></a>Increment | The usable, potentially releasable piece of software created during a sprint. | "The team successfully delivered a working increment of the new dashboard." |
| <a id="user-story"></a>User Story | A short, simple description of a feature told from the perspective of the user. | "Can you write a User Story for the password reset functionality?" |
| <a id="ac"></a>Acceptance Criteria (AC) | The conditions that a user story must meet to be considered "done." | "The QA team can't test this until the Acceptance Criteria are clearly defined." |
| <a id="dod"></a>Definition of Done (DoD) | The team's shared understanding of what it means for a piece of work to be complete. | "According to our Definition of Done, all code must be peer-reviewed and have 80% test coverage." |
| <a id="velocity"></a>Velocity | The average amount of work a team can complete in a single sprint. | "Our team's velocity is about 25 story points per sprint." |
| <a id="story-points"></a>Story Points | A relative unit of measure to estimate the effort required to complete a user story. | "This story feels like 8 Story Points because of the database complexity." |
| <a id="fibonacci"></a>Fibonacci Sequence | The sequence of numbers often used for story point estimation (1, 2, 3, 5, 8, 13...). | "Remember team, we use the Fibonacci sequence for our estimations, so don't pick a 4." |
| <a id="planning-poker"></a>Planning Poker | A consensus-based technique for estimating story points. | "Let's use Planning Poker to estimate the effort for the checkout feature." |
| <a id="epic"></a>Epic | A large user story that can be broken down into smaller stories. | "We need to break down the 'User Profile Redesign' Epic into smaller, manageable stories." |
| <a id="theme"></a>Theme | A collection of related user stories. | "This sprint's theme is 'Performance Improvements'." |
| <a id="spike"></a>Spike | A time-boxed research task to investigate a technical question or uncertainty. | "I'm creating a Spike to investigate which third-party payment API we should use." |
| <a id="impediment"></a>Impediment / Blocker | Anything that prevents the team from making progress. | "I have an impediment; my developer environment isn't working." |
| <a id="time-box"></a>Time-box | A fixed period of time for an activity (e.g., a meeting or a sprint). | "Let's time-box this discussion to 10 minutes to keep the meeting on track." |
| <a id="burndown"></a>Burndown Chart | A graph showing the amount of work remaining in a sprint. | "The Burndown Chart shows we are tracking behind schedule for this sprint." |
| <a id="burnup"></a>Burn-up Chart | A graph showing the amount of work completed over time. | "Looking at the Burn-up Chart, we can see the total scope increased mid-sprint." |
| <a id="kanban"></a>Kanban | A visual workflow management method, often used with Scrum (Scrumban). | "Our support team uses a Kanban board to manage incoming tickets." |
| <a id="wip"></a>WIP (Work in Progress) | The number of tasks being worked on at any given time. | "We need to lower our WIP limit to improve focus and get stories to 'Done' faster." |
| <a id="cycle-time"></a>Cycle Time | The time it takes to complete a task from start to finish. | "Our average Cycle Time for a bug fix has decreased from 3 days to 2." |
| <a id="lead-time"></a>Lead Time | The time from when a task is requested to when it is delivered. | "The Lead Time for new feature requests is currently about two months." |
| <a id="tech-debt"></a>Technical Debt | The implied cost of rework caused by choosing an easy solution now instead of a better approach. | "That quick fix added more technical debt that we'll have to address later." |
| <a id="refactoring"></a>Refactoring | Improving the internal structure of code without changing its external behavior. | "We should dedicate the next sprint to refactoring the old authentication module." |
| <a id="swarming"></a>Swarming | When the whole team focuses on a single user story to get it done faster. | "This is a critical story. Let's try swarming on it to ensure it's done by tomorrow." |
| <a id="walking-board"></a>Walking the Board | Going through the tasks on the Scrum/Kanban board during a stand-up. | "Instead of going person by person, let's start walking the board from right to left." |
